What we're looking at

This dataset covers a slice of federal IT procurement: specifically NAICS 541511 (Custom Computer Programming) and 541512 (Computer Systems Design) from FY2022 through FY2026. These are two codes within the broader 5415 "Computer Systems Design and Related Services" family, which also includes 541513 (Computer Facilities Management), 541519 (Other Computer Related Services), and others. We picked these two as a starting sample — the pipeline can be extended to other NAICS codes.

The contract records come from USASpending bulk award archives. All code is on GitHub and the pipeline is designed to be extended to other NAICS codes.

How are these contracts evaluated?

We classified every contract by how it was competed, combining two data sources: FPDS tradeoff codes (via the Tango API) tell us whether a contract used LPTA or best-value tradeoff evaluation, and USASpending competition fields tell us the solicitation method for everything else.

What fields do we have to work with?

The USASpending bulk data has ~75 fields per transaction. Here are the ones that seem most interesting for understanding how IT services are bought. The field names link to their definitions in the USASpending data dictionary.

FieldWhat it tells youNotes
extent_competed_code Whether the contract was competed, and how A = full & open, D = after exclusion, F/G = SAP, B/C = not competed
solicitation_procedures_code How offers were solicited NP = negotiated proposal, MAFO = fair opportunity (IDIQ), SSS = sole source, SP1 = simplified
number_of_offers_received How many vendors bid Often blank or "999" (unknown). When populated, shows competition intensity
type_of_contract_pricing_code Fixed price vs. cost-reimbursable vs. T&M J = FFP, Y = T&M, Z = labor hours, K = FP with EPA
type_of_set_aside_code Small business set-aside type SBA, 8A, SDVOSBC, WOSB, HZC, etc. or NONE
performance_based_service_acquisition_code Whether the contract uses performance-based criteria ~60% of IT contracts are marked performance-based (Y)
commercial_item_acquisition_procedures_code Whether FAR Part 12 commercial procedures were used ~50% use commercial item procedures
fair_opportunity_limited_sources_code For IDIQ task orders: was fair opportunity given? Distinguishes competed task orders from sole-source task orders
cost_or_pricing_data_code Whether certified cost/pricing data was required Relates to Truth in Negotiations Act (TINA) thresholds
contracting_officers_determination_of_business_size_code CO's determination: small or other than small S = small, O = other than small. Independent of set-aside
product_or_service_code What specifically was bought (more granular than NAICS) D-series = IT services. Could sub-categorize by service type
solicitation_identifier Links back to the original solicitation Could be used to find the actual RFP on SAM.gov or beta.SAM.gov

We also pull source_selection_process (LPTA vs. tradeoff) from FPDS via the Tango API — this field isn't in USASpending. With SAM.gov enrichment, we can add vendor age, employee count, and SBA certifications.

What we can't tell yet

Most contracts don't have LPTA/tradeoff data. The FPDS source_selection_process field is contractor-reported and blank for ~40–60% of awards. We currently have tradeoff codes for about 13% of contracts. The rest are classified using USASpending competition fields, which tell us how a contract was competed but not the specific evaluation criteria.

"Negotiated Proposal" is a catch-all. It captures all full-and-open competitions that aren't LPTA, tradeoff, or fair opportunity. This is 43% of all contracts — there's almost certainly interesting variation hiding inside this category that we can't see from the data we have.

Fair Opportunity dominates by dollars. IDIQ/GWAC task orders account for most of the money but are evaluated differently from standalone contracts. The evaluation process happens at the task-order level and the details aren't captured in these fields.

Questions this raises

This is a work in progress. Some things I'd like feedback on:

  • Is the evaluation method classification reasonable? Are there better ways to cut this using fields available in FPDS/USASpending?
  • For "Negotiated Proposal" contracts — is there a way to distinguish best-value from LPTA when the tradeoff_process field is blank?
  • How should we think about fair opportunity task orders vs. standalone competed contracts? Are they meaningfully comparable?
  • What would be the most useful next analysis to run on this data?

If you have ideas or spot something wrong, open an issue or reach out. The full pipeline is open source.

Methodology

1
Contract data — USASpending bulk archives

Transaction-level records from USASpending award data archives, filtered to NAICS 541511 + 541512. Aggregated to one row per contract using the latest total_dollars_obligated (cumulative) and latest categorical values.

Source: fetch_bulk.py · build_contracts.py

▼
2
LPTA/tradeoff codes — Tango API (FPDS)

USASpending doesn't publish source_selection_process. We pull it from FPDS via the Tango API and join on contract_award_unique_key. Currently matched to ~13% of contracts.

Source: fetch_tradeoff.py

▼
3
Evaluation method classification

Each contract is classified into one category. Tango codes take priority when available; USASpending competition fields classify the rest.

Category Rule Source
LPTA tradeoff_process.code = "LPTA" Tango (FPDS)
Best-Value Tradeoff tradeoff_process.code = "TO" Tango (FPDS)
Fair Opportunity solicitation_procedures_code = "MAFO" USASpending
Negotiated Proposal extent_competed in (A, D) AND solicitation_procedures = "NP" USASpending
Simplified Acquisition extent_competed in (F, G) USASpending
Sole Source solicitation_procedures = "SSS" USASpending
Not Competed extent_competed in (B, C), not sole source USASpending

Source: build_contracts.py — classify_eval_method()
